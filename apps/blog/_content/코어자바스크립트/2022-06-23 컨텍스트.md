## 01. 실행컨텍스트

**실행 컨텍스트**란 무엇인가?

> 실행할 코드에 제공할 환경 정보들을 모아놓은 객체로, 자바스크립트의 동적 언오러서의 성격을 파악할 수 있는 개념

**스택**은 무엇인가?

> 출입구가 하나 뿐인 깊은 우물같은 데이터 구조. 데이터를 100개만 담을 수 있는 우물에 100개 이상의 데이터를 넣으려고하면 넘치듯, 많은 언어들이 스택이 넘칠 때 에러를 던진다.

**큐**는 무엇인가?

> 양쪽이 모두 열려있는 파이프 형태. 종류에 따라 양쪽에 입력과 출력이 가능한 큐도 있으나, 보통은 한쪽엔 입력만, 다른 한쪽은 출력을 담당하는 구조다.

---

1. 실행 컨텍스트는 동일한 환경에 있는 코드들을 실행할대 필요한 환경 정보를 모아 컨텍스트를 구성한다.

2. 이를 콜 스택에 쌓아올린 후, 가장 위에 쌓여있는 컨텍스트와 관련 있는 코드들을 시행하는 방식으로 전체 코드의 환경과 순서를 보장한다.

여기서 말하는 **동일한 환경**이란, 총 4가지 타입으로 나눠지는데, 이 소스코드에 따라 컨텍스트를 생성하게 된다. 간단하게 4가지 타입에 대해 정리하자면 다음과 같다.

### 전역코드

정의

- 전역에 존재하는 소스코드, 전역에 정의된 함수, 클래스 등의 내부코드는 포함하지 않는다.

생성, 관리 과정

- 전역 코드는 전역 변수를 관리하기 위해 최상위 스코프인 전역 스코프를 생성해야한다.
- var 키워드로 선언된 전역 변수와 함수 선언문으로 정의된 전역 함수를 전역 객체의 프로퍼티와 메서드로 바인딩하고 참조하기 위해 전역 객체와 연결되어야 한다.

### eval 함수코드

정의

- 빌트인 전역 함수인 eval 함수에 인수로 전달되어 실행되는 소스코드

생성, 관리 과정

- strict 모드에서 자신만의 독자적인 스코프를 생성한다.

### 함수코드

정의

- 함수 내부에 존재하는 소스코드, 함수 내부에 2중첩된 함수, 클래스 등의 내부 코드는 포함하지 않음

생성, 관리 과정

- 지역 스코프를 생성하고 지역 변수, 매개변수, arguments 객체를 관리해야 한다. 생성한 지역 스코프를 전역 스코프에서 시작하는 스코프 체인의 일원으로 연결해야 한다.

### 모듈코드

정의

- 모듈 내부에 존재하는 소스코드

생성, 관리 과정

- 모듈별로 독립적인 모듈 스코프를 생성한다.

```js
let a = 1;

function outer() {
  function inner() {
    console.log(a); // undefined
    let a = 3;
  }
  inner();
  console.log(a, 'outer');
}
outer(); // 제일 먼저 실행, 1
console.log(a); // 두번째 실행 , 1
```

> 1. 전역 컨텍스트와 관련된 정보를 수집, 브라우저가 열리는 순간 활성화
> 2. 순차 진행하다가 outer함수 호출하며 함수 환경정보 수집 후, 콜 스택에 담음 (콜스택 최상위엔 현재 outer 실행 컨텍스트가 있음)
> 3. 최상단에 outer 실행 컨텍스트가 놓였으므로, 전역 컨텍스트와 관련된 실행을 일시 중단하고, outer 함수 내부의 코드를 순차 실행
> 4. 내부에 inner 함수가 발견됐으므로 최상단 콜스택에 inner 컨텍스트가 쌓이게 됨
> 5. 이제 최상단에 놓여진 컨텍스트부터 순차 실행하게 됨
> 6. 결론 // undefined, (1,"outer"), 1 이 실행됨

---

스택 구조를 생각해보면 컨텍스트가 콜 스택의 맨 위에 쌓이는 순간이 곧 현재 실행할 코드에 관여하게 되는 시점임을 알 수 있다. 기존의 컨텍스트는 새로 쌓인 컨텍스트보다 아래에 위치할 수 밖에 없다.

자바스크립트 엔진은 해당 컨텍스트와 관련된 코드를 실행할대 필요한 환경정보를 수집해서 객체로 담아 실행 컨텍스트에 저장하고, 이 객체는 엔진이 활용할 목적으로 생성할 뿐, 개발자가 확인할 순 없다.

여기에 담기는 정보는 다음과 같다.

1. `VariableEnvironment` : 현재 컨텍스트 내의 식별자들에 대한 정보와 외부 환경정보, 선언 시점의 렉시컬 환경의 스냅샷으로, 변경사항이 반영되지 않는다.
2. `LexicalEnvironment` : 처음에는 베리어블 환경과 같지만 변경사항이 실시간으로 반영됨
3. `ThisBinding` : this 식별자가 바라봐야 할 대상 객체

## 02. VariableEnvironment

실행 컨텍스트 생성시 `VariableEnvironment` 에 정보를 담은 다음, 이를 복사해서 렉시컬 환경을 만들고 이후에는 렉시컬을 주로 활용한다.

`VariableEnvironment`와 `LexicalEnvironment`의 내부는 `environmentRecord`와 `outer-EnvironmentReference`로 구성돼있다. 초기화 과정 중에는 사실상 완전히 동일하고 이후 코드 진행에 따라 서로 달라지게 된다.

## 03. LexicalEnvironment

`LexicalEnvironment`은 어휘적 환경말로 이해할 수 있다. 쉽게 말해 `현재 컨텍스트의 내부에는 ~~이러이러한 시결자들이 있고 그 외부 정보는 ~~를 참조해라` 라는 방식의 정보가 담겨있다. 컨텍스트들을 구성하는 환경 정보들을 사전에 접하는 느낌으로 모아둔 것이다.

### 2-3-1 environmentRecord와 호스팅

`environmentRecord`에는 현재 컨텍스트와 관련된 코드의 식별자 정보들이 저장된다. 컨텍스트를 구성하는 함수에 지정된 매개변수 식별자, 선언한 함수가 있을 경우 그 함수 자체, var로 선언된 변수의 식별자등이 해당한다. 컨텍스트 내부 전체를 처음부터 훑어나가며 순서대로 수집한다.

변수 정보를 수집하는 과정을 마쳤어도, 컨텍스트가 관여할 코드들은 실행되기 `이전`이다. 실행이 되지 않았음에도 자바스크립트 엔진은 **이미 해당 환경에 속한 코드의 변수명들을 모두 알고 있게 된다.**

그래서 여기서 호이스팅이란 개념이 적용된다. 실행되기 이전에 모두 알고있으니까 이는 곧 식별자들을 최상단으로 **끌어올린 다음에 실행해도 문제가 없다**는 뜻이다. 뭐 실제로 끌어올리진 않지만, 우리가 이해하기 쉽게 하기 위해서 끌어올린 것으로 간주하는 것이다.

### 함수 선언문과 표현식

함수 선언문과 표현식에 대해서도 알아두자. 둘 모두 함수를 새롭게 정의할 때 쓰이는 방식으로

함수 선언문은 function 정의부만 존재하고 별도의 할당 명령이 없는 것을 의미한다.

함수 표현식은 정의한 function을 별도의 변수에 할당하는 것을 의미한다.

```js
function a() {} //함수 선언문
a();
var b = function () {}; // 익명 함수 표현식
b();
var c = function d() {}; // 기명 함수 표현식
c();
d(); // error!
```

선언식과 표현식은 이후 호이스팅에서 차이를 보인다. 엔진이 선언문은 함수 자체를 호이스팅하는 반면, 표현식은 변수 부분만 호이스팅이 되기 때문이다.

```js
console.log(sum(1, 2));
console.log(multiply(3, 4)); //muliply is not a function

function sum(a, b) {
  return a + b;
}

var multiply = function (a, b) {
  return a * b;
};
```

여기서 sum 함수 선언문은 정상적으로 작동한다. 그러나 multiply는 함수가 아니라며 에러가 발생한다.

내부 실행 방식을 뜯어보면 그 이유를 알 수 있다.

> 1. 메모리공간 확보 후 주솟값을 변수 sum에 연결한다.
> 2. 또 다른 메모리 공간을 확보 후 변수 multiply에 연결한다.
> 3. sum 함수를 다른 메모리 공간에 저장하고, 그 주솟값을 앞서 선언한 변수 sum의 공간에 할당한다. 이로써 sum은 함수 sum을 바라보는 상태가 된다.
> 4. multply는 값이 할당돼있지 않다. 비어있는 대상을 함수로 여겨 실행하라고 명령해서 오류가 난 것이다.

함수 선언문은 전역에서 실행될때, 제일 마지막에 할당된 함수가 전역에 덮어씌워진다. 그래서 협업시 같은 이름의 함수 선언식을 사용하게 된다면, 제일 마지막에 선언된 함수가 이전의 값을 모두 대체해버리는 대참사가 발생한다.

이럴때 표현식을 사용하면, 함수를 작성한 라인 이후부터 적용됨으로 의도한대로 작동하게 된다.

## 2-3-2 스코프, 스코프 체인, outerEnviromnetReference

스코프란 식별자에 대한 유효범위입니다. a의 외부에서 선언한 변수는 a 내외부에서 접근이 가능하지만, a내부에서 선언한 변수는 오직 내부에서만 접근 가능하다.

이렇게 식별자의 유효범위를 안에서부터 외부로 차례로 검색해 나가는 것을 `스코프 체인`이라고 부른다. 이를 가능케하는 것이 렉시컬 환경의 두번째 수집자료인 `outerEnviromnetReference` 덕분이다.

### 스코프 체인

`outerEnviromnetReference`는 현재 호출된 함수가 `선언될 당시`의 렉시컬 환경을 참조한다. 즉 선언하다는 행위는 콜스택상에서 어떤 실행 컨텍스트가 활성화 된 상태일 뿐이다. 선언하는 행위 자체도 하나의 코드이며, 모든 코드는 실행 컨텍스트가 활성화 상태일때 실행된다는 것을 잊지말자.

`outerEnviromnetReference`는 연결리스트의 형태를 띄는데, 이를 알아보기 위해선 간단한 예시를 들어보자.

A함수 내부에 B라는 함수를 선언하고, 다시 B함수 내부엔 C함수를 선언한 경우, 함수 C의 `outerEnviromnetReference`는 함수 B의 렉시컬 환경을 참조한다.

함수 B의 렉시컬 환경에 있는 `outerEnviromnetReference`는 다시 함수 B가 선언된 A의 렉시컬 환경을 참조하는 형태다. 서로 꼬리에 꼬리를 무는 형태를 두고 연결 리스트라고 부른다.

이렇게 선언시점의 렉시컬 환경을 계속 찾아올라간다면, 마지막엔 전역 컨텍스트의 렉시컬 환경이 있을 것이다. 이런 구조적 특성 덕분에 `outerEnviromnetReference`은 자신이 선언된 시점의 렉시컬환경만 참조하고 있으므로 가장 가까운 요소부터 차례대로만 접근할 수 있다.
(마치 리액트에서 props drilling과 유사한 느낌이 나기도 한다.)

이런 특성덕에, 여러 스코프에서 동일한 식별자를 선언한 경우 무조건 스코프 체인상에서 가장 먼저 발견된 식별자에만 접근 가능하게 된다.

## 정리

실행 컨텍스트는 실행할 코드에 제공할 환경 정보를 모아둔 객체다.

실행 컨텍스트는 전역 공간에서 자동으로 생성되는 전역 컨텍스트와 eval함수, 함수 실행에 의한 컨텍스트 등이 있다. 실행 컨텍스트 객체는 활성화되는 시점에 variableEnvironment, LexicalEnvrionment, ThisBinding 세가지 정보를 수집한다.

실행 컨텍스트를 생성할땐 베리어블과 렉시컬이 동일한 내용으로 만들어지지만, 렉시컬은 함수 실행 도중에 변경되는 사항이 즉시 반영되는 반면, 베리어블은 초기상태를 유지한다.
