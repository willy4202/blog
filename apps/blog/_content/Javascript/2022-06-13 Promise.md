프로젝트를 진행하며 가장 많이 마주친 에러가 있다면 `Promise-reject`일 것이다. 약속이 거절됐다고? 도대체 무슨 말인가 이게,,

안그래도 머릿속에 들어오는 개념이 많아서 혼란스러웠는데, 이번 기회에 Promise를 확 잡고 가야겠다.

## 배경

ES6에선`Promsie`를 통해, 콜백 패턴이 가진단점을 보완하며, 비동기 처리 시점을 명확하게 표현할 수 있다는 장점이 있어서 혁신이라 불리기도 했었다.

여기서 말하는 콜백 패턴의 단점을 간략하게 짚어보자면 콜백 헬이 있다.

이를 이해하려면, 동기식, 비동기식 처리 모델에 대해서도 알아야한다. 동기식 처리 모델은 순서대로 처리되는 태스크로, 서버에 요청을 전달하고 응답될때까지 다른 태스크를 진행하지 않는다.

비동기식 처리 모델은 병렬적으로 태스크를 수행한다. 서버에 데이터를 요청한 이후 서버로부터 데이터가 응답될때까지 대기하지 않고, 다음 태스크를 수행한다. 이후 서버로부터 응답이 오면 그에 상응하는 태스크가 진행되는 방식이다.

대표적으로`Dom`,`Timer`,`Ajax`요청이 비동기식 모델이다.

비동기 처리 모델을 잘 활요하면 작업이 중단되지 않는 장점이 있다. 다만, 콜백 패턴을 사용하면 처리 순서를 보장하기 위해 콜백 함수를 중첩해서 사용하는데, 이때 복잡도가 점점 높아지는 현상을 콜백헬이라고 한다.

이는 가독성을 해치며, 실수를 유발하게 만드는 원인이된다. 또한, 에러 처리도 어려워진다.

## Promise란?

`Promise`는 비동기 동작을 처리하기 위해 ES6에 도입됐다. 엄밀히 말하자면 클래스이지만 클래스를 인스턴스화 해서 `Promise` 객체를 만든다고 한다.

이렇게 사용한다면 비동기 메서드에서 마치 동기 메서드로 값을 반환할 수 있다. 다만 최종 결과를 반환하는 것이 아니라 결과를 제공하겠다는 약속`Promise`을 반환하는 것이다.

#### 그래서 이게 왜필요하냐?

라고 물어볼 수 있다. 결국 코드를 사용하는 이유는 다음과 같다.

> 프로미스는 주로 서버에서 받아온 데이터를 화면에 표시할 때 사용한다. 일반적으로 웹 애플리케이션을 구현할 때 서버에서 데이터를 요청하고 받아오기 위해서다.

즉, 데이터를 모두 받아오기도 전에, 다 받아온것마냥 화면에 데이터를 표시하려고 하면 오류가 발생하거나, 빈 화면이 뜨는 것이다.

내가 겪은 에러도 비슷한 경우였다.

> `Promise`는 생성된 시점에는 알려지지 않았을 수 있는 값을 대신한다. 즉, "값이 없을때 이녀석이 대신할거야~, 대신! 값이 나왔을땐 내가 그 녀석을 반환할게~" 라는 의미로 생각하면 된다.

이런 `Promsie`는 두가지 인자를 전달 받는데, `resolve`와 `reject`이고, 가질 수 있는 상태는 다음과 같다.

대기(pending): 이행하지도, 거부하지도 않은 초기 상태.
이행(fulfilled): 연산이 성공적으로 완료됨.
거부(rejected): 연산이 실패함.

## 구현하기

```js
let promise = new Promise(function (resolve, reject) {
  // 여기 비동기 로직을 작성!
  // 시켜놓고 언제 완료될지 모르는 로직!
});
```

resolve 는 비동기 로직이 성공했을 경우 실행하는 함수
reject는 비동기 로직이 실패했을 경우다.

간단한 예제 코드로 promise의 동작에 대해 더 알아보자

```js
function getData() {
  return new Promise(function (resolve, reject) {
    $.get('url 주소/products/1', function (response) {
      if (response) {
        resolve(response);
      }
      reject(new Error('Request is failed'));
    });
  });
}

// 위 $.get() 호출 결과에 따라 'response' 또는 'Error' 출력
getData()
  .then(function (data) {
    console.log(data); // response 값 출력
  })
  .catch(function (err) {
    console.error(err); // Error 출력
  });
```

위 코드는 제대로 응답을 받아오면 resolve()를, 아니라면 reject()를 호출한다. 호출된 메서드에 따라 then으로 후속 응답을 처리하는 방식이다.

## 에러

여기서 궁금한게 생긴다. then은 알겠는데,,, `catch`는 무엇인가? then은 이후에 처리하는 로직인 것 같은데, 에러를 어떻게 구분할 수 있을까에 대한 의문이 많았다.

현업에서는 서비스를 구현하다보면 네트워크 연결, 서버 문제 등으로 인해 오류가 발생핤 수 있다고 한다. 그래서 에러 처리를 2가지 방식으로 할 수 있다고 한다.

#### then()의 두번째 인자로 에러 처리

```js
getData().then(handleSuccess, handleError);
```

#### catch() 이용

```js
getData().then().catch();
```

위 와 같은 방법으로 reject() 메서드가 호출되었을때 에러를 핸들링할 수 있다.

이 두가지는 언제 사용하면 좋을까? 구글링을 해보니 가급적이면 catch를 추천한다고 한다. 그 이유는 더 정확한 디버깅을 위해서라고 한다.
